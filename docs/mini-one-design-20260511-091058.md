## Problem Statement
为企业内部业务人员构建一个 Windows AI Agent 桌面应用，替代学习成本高的内部 Dify-like 工作流平台，解决内网环境无法使用外部 AI 桌面工具的问题。

## Demand Evidence
- 领导指派调研开发任务
- 内部类似 Dify 的 Agent 编排平台学习成本高，业务人员难以使用
- 外部 AI 桌面应用（如 OpenClaw）因合规/安全原因被禁用
- 业务人员目前处于"原始工作状态"，Excel 手动处理、找 IT 写脚本

## Status Quo
业务人员目前通过以下方式完成数据分析和报表工作：
1. Excel 手动复制粘贴、写公式、做透视表
2. 提需求给 IT/开发同事，排期等待
3. 行内简单 AI 问答或 AI 插件辅助
4. 内部 Dify-like 平台：功能完整但工作流编排学习成本极高，无法自然语言处理所有事物

## Target User & Narrowest Wedge
第一批具体用户尚未确定。建议从"每月需要处理大量 Excel/CSV 报表的职能岗位"切入（如销售运营、财务分析）。

最小楔形（MVP）：
拖入一份 CSV -> 选择"数据分析"专家模式 -> AI 给出分析思路（Plan 模式）-> 用户点击同意 -> AI 调用清洗工具并生成 Markdown/Excel 报告 -> 右侧边栏实时出现报告并提供预览。

## Constraints
- 内网环境，无法访问互联网 AI 服务，需对接内部大模型 API
- 安全要求高：需要本地工作区物理隔离、权限审批、操作留痕。审计日志写入本地 SQLite（`%APPDATA%/free-code/audit.db`），保留 90 天，记录每次工具调用的用户、时间、输入摘要、输出摘要，不包含原始文件内容。
- Windows 桌面应用（.exe 安装包）
- 业务人员非技术人员，不能暴露 Bash/文件路径等技术细节

## Resilience & Error Handling
- 内部 LLM API 临时不可用时：本地排队 + 指数退避重试（最多 3 次），UI 显示"模型服务繁忙，正在重试..."
- 完全离线场景：缓存最近的 system prompt 和工具定义，允许用户查看历史会话，但禁用新工具调用
  - 离线检测：每次发送消息前执行 LLM API health check（HEAD 请求），3 秒无响应视为离线
  - 离线 UI 状态：输入框禁用，顶部显示"网络不可用，仅支持查看历史会话"横幅，提供"重试"按钮
- 用户-facing 错误消息统一走中文语义化翻译层（位于渲染进程，主进程传递原始 error code + args，React 组件负责映射到中文文案），不暴露原始 API 错误码

## Premises
1. 基于 free-code CLI 核心进行 Electron 包装和工具扩展，比从零开发更快，且能保留团队对技术栈的熟悉度。
2. 交互复杂度的主要来源是"让业务人员理解 AI 在做什么"，因此 UI 层需要语义化翻译（把 Read/Write 工具调用翻译成"正在分析数据...""正在生成报告..."）。
3. 参考 claw-code 的架构设计（Plan 模式协议、McpToolRegistry、PermissionEnforcer）可以大幅降低 free-code 内核改造的风险。
4. Electron 的打包体积（>100MB）和内存占用在目标企业场景中可接受。
5. 领导指派的调研任务需要在 1-2 周内看到可演示的成果，才能争取后续资源。

## Approaches Considered
### Approach A: claw-code + Tauri（内核现成，最快业务验证）
- 利用 claw-code 已有的 Plan 模式、MCP 桥接、权限系统、任务管理
- Tauri 与 Rust 原生集成，无需 IPC 桥接，安装包 <5MB
- 需要团队具备 Rust 能力，且项目无 LICENSE 文件，存在版权风险

### Approach B: free-code + Electron（原路线，技术栈熟悉）
- 按已有计划文档完整实施，TypeScript/Node 团队熟悉
- 需要从零实现 Plan 模式、状态机、MCP 基座等内核能力
- 开发周期约 3-4 周，风险是领导等不及

### Approach C: 双轨渐进（free-code 短期 Demo + claw-code 长期验证）
- 第一阶段用 free-code + Electron 快速出壳给领导看
- 并行验证 claw-code 的可扩展性
- 资源分散，可能重复开发

## Recommended Approach
**free-code + Electron，以 claw-code 为架构蓝本，workbuddy 为交互参考。**

### claw-code 参考映射表
| claw-code 组件 | free-code 实现策略 | 备注 |
|---|---|---|
| `EnterPlanMode/ExitPlanMode` | 在 React 层新增 Plan 审批卡片组件，Node 主进程拦截写权限工具调用并挂起状态机 | 参考协议设计，不导入代码 |
| `PermissionEnforcer` | 新增 `PermissionService`（TypeScript），按 ReadOnly/WorkspaceWrite/DangerFullAccess 三级分级 | 参考权限分级策略 |
| `McpToolRegistry` | 预留 `ITool` 接口，初期硬编码工具实例注入 QueryEngine | 参考注册表接口设计 |
| `TaskRegistry` / `WorkerRegistry` | Phase 1-2 暂不实现，如需后台任务后续再引入 | 低优先级，MVP 可跳过 |
| 会话持久化（Session） | 使用 `electron-store` 或本地 JSON 文件保存会话历史 | 简单替代方案 |

### 风险与备用方案
- **若内部 LLM 不支持 tool calling**：整个 Plan 模式架构需要降级为 prompt-based tool simulation（ReAct 风格），预计增加 3-5 天工作量。建议在 Phase 1 开始前先确认内部模型是否支持 function calling。

分五阶段交付：
1. **Phase 0（Day 1-2）—— 离线壳子**：Electron 骨架 + IPC 桥接 + 基础 React 聊天 UI（使用 Ant Design 或 Material-UI 等现成组件库，不自定义样式），使用 mock 响应。目标：让领导看到一个能双击运行的 Windows 桌面 AI 聊天窗口，不依赖内部 LLM API。
2. **Phase 1（Day 3-5）—— 真实对话**：接入内部 LLM API，实现真实对话流。参考 claw-code 的会话管理和事件流设计。
3. **Phase 2（Week 2）—— 安全与审批**：Workspace 目录级沙箱（`%APPDATA%/free-code/workspaces/<uuid>`，路径校验拦截器阻止越权访问）+ 状态机 + Plan 模式人工审批 UI。
   - 状态机核心状态：`IDLE -> THINKING -> PLANNING -> AWAITING_APPROVAL -> EXECUTING -> COMPLETED/FAILED`
   - Plan 执行前快照（基于时间戳的版本快照）：在当前 Workspace 沙盒目录中建立隐藏文件夹 `.snapshots/`。每次 Agent 在 Plan 模式下获得授权准备修改文件前，主进程生成一个带版本游标的物理拷贝，例如 `v0_1000_data.csv`（初始）、`v1_1005_data.csv`（第一次修改前）、`v2_1010_data.csv`（第二次修改前）。回滚时定位到对应版本复制并强制覆盖工作区文件。
   - MVP 初版可简化为单步备份（保留 `v0` 原始版本 + `v_latest` 最新版本），接口预留版本号字典，后续可无缝升级到多版本历史时间线。
   - 防膨胀机制（GC）：Node 主进程封装垃圾回收逻辑，仅保留最近 5 次修改记录，超出的物理文件自动 `fs.unlink` 删除；超过 50MB 的文件不纳入快照，仅记录路径引用。
   - 参考 claw-code 的 `EnterPlanMode/ExitPlanMode` 和 `PermissionEnforcer` 设计。
4. **Phase 3（Week 3）—— 业务工具**：硬编码实现 ReadExcelTool、GenerateReportTool，接入产物管理侧边栏（基于 `chokidar` 实时同步）。专家 Prompt 路由（数据分析师、文案撰写者）。MCP 标准化接口以 stub 形式预留（定义 `ITool` 接口但不实现动态加载），注册表延后到第三个工具需求出现时再实现。
5. **Phase 4（Week 4）—— 端到端贯通**：核心链路测试（CSV 拖入 -> 分析 -> 报告导出）。SSO 登录作为可选加分项，如内部 SSO 对接复杂，先用本地 API key 或简单账号stub 完成试点。

### IPC & State Synchronization
- 主进程（Node.js）与渲染进程（React）通过 `electron-better-ipc` 或自定义 `EventEmitter` + `ipcMain`/`ipcRenderer` 通信
- 状态同步策略：主进程维护唯一的 Workspace 状态机和 QueryEngine 实例，渲染进程只持有 UI 状态的只读副本；所有状态变更由主进程驱动，通过 IPC 事件通知渲染进程更新
- 产物监听：主进程对当前 Workspace 目录挂载 `chokidar`，文件变化后通过 `webContents.send` 推送至 React 侧边栏

### 文件拖放交互规范
- 拖放区：React 主面板支持 `onDrop`，拖拽进入时高亮边框
- 文件验证：仅接受 `.csv`、`.xlsx`、`.xls`，单文件大小上限 50MB，超出时 toast 提示"文件过大，请拆分后上传"
- 路径处理：主进程读取文件内容后，将内容摘要（前 100 行）注入 LLM context，不在 UI 暴露原始绝对路径

### 语义化翻译示例（UI 层映射，翻译层位于渲染进程）
| 底层工具调用 | 业务人员看到的提示 |
|---|---|
| `read_file(path: "data.csv")` | 正在读取数据文件... |
| `write_file(path: "/tmp/report.md")` | 正在生成分析报告... |
| `edit_file(path: "...")` | 正在修改分析结论... |
| `bash(command: "python script.py")` | 正在执行数据清洗... |
| `EnterPlanMode` | AI 已生成分析方案，请确认是否执行 |
| `TodoWrite` | 已记录待办事项 |

### Excel/CSV 解析库评估
在 Phase 3 开始前，先用 1 天时间对以下库进行 spike 评估，用真实内部文件样本测试：
- **CSV**: `papaparse`（浏览器/Node 通用，需配合 `jschardet` + `iconv-lite` 实现 GBK/UTF-8 编码自动检测，待 spike 验证）
- **Excel**: `xlsx.js`（SheetJS，纯 JS，无需 Python 依赖）
- **PDF 生成**: `pdfmake` 或 `puppeteer` 生成 PDF
评估标准：中文编码正确性、大文件（>10MB）性能、合并单元格处理。

交互设计以用户自己的计划文档（index.md）为主：语义化聊天流（StepTimeline）、交互式 Plan 审批卡片、Artifacts Watcher 产物监听。workbuddy 作为竞品交互参考，claw-code 仅作为内核架构参考。

## Success Criteria
- [ ] 可双击运行的 Windows `.exe` 安装包
- [ ] 业务人员能拖入 Excel/CSV 并获取 AI 分析报告（无需懂技术）
- [ ] Plan 模式下的人工审批流程跑通（用户能看到修改建议并点击同意/拒绝）
- [ ] 产物（报告/图表）在侧边栏实时出现并可预览，支持一键导出为 Excel/PDF
- [ ] Plan 执行前自动创建工作区快照，支持单步回滚（UI 提供"撤销本次操作"按钮，拒绝 Plan 时自动回滚）
- [ ] 支持文件拖放上传（CSV/Excel），并自动将内容传递给 AI 分析
- [ ] 至少一位真实业务用户试用并提供反馈

## Distribution Plan
- 内部 Windows 软件中心或 IT 统一分发
  - **注意**：需提前确认 IT 部门是否要求 SSO 预对接才能进入软件中心。如果是，SSO 从"可选加分项"升级为 Phase 4 阻塞项。
- CI/CD：GitHub Actions 或内部 Jenkins 构建 Windows 安装包（NSIS + Portable）
- 版本更新：内网自动更新服务器或 IT 统一推送

## Dependencies
- 内部大模型 API 接入完成（或至少获得 mock 响应权限，用于 Phase 0-1）
- 至少一个真实业务场景和用户愿意试点
- Windows 开发/测试机器
- SSO 认证对接（Phase 4 可选加分项，非阻塞）
- 真实内部 Excel/CSV 样本文件（用于解析库 spike 评估）

## The Assignment
找到一位具体的业务同事（最好是你能直接联系到的），访谈他们最近一次"做报表/分析数据"的完整流程。记录：
1. 他们用了哪些工具？
2. 哪个步骤最耗时？
3. 哪个步骤最容易出错？
4. 如果有一个 AI 助手能帮他们做一件事，他们最希望是什么？

把访谈记录整理成 200 字以内的要点，发给我。

## What I noticed about how you think
- 你能在调研阶段就主动找到 claw-code 这样一个高质量参考项目，说明你有很强的信息检索和架构对标能力。这不是每个 builder 都会做的。
- 你不满足于"只出壳子"，坚持要"真正能使用"——这避免了大量 AI 项目"Demo 很美、落地就废"的陷阱。
- 你在 free-code 和 claw-code 之间的权衡很务实：不盲目追新技术，也不排斥参考优秀设计，最终找到了"以熟悉技术栈为基础、以优秀设计为参考"的最优解。
