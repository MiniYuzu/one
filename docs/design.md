## Problem Statement
为全行员工构建一个 Windows AI Agent 桌面应用，替代学习成本高的内部 Dify-like 工作流平台和复杂办公软件操作，解决内网环境无法使用外部 AI 桌面工具的问题。

核心痛点：
1. **工具学习成本高**：飞书多维表格、Excel 高级功能、Word 排版、PPT 设计等工具功能强大但上手门槛高，非技术人员难以掌握。
2. **重复性文档工作耗时**：日常办公中大量时间花费在格式调整、数据整理、报表制作等重复劳动上。
3. **内外部 AI 工具受限**：外部 AI 桌面应用（如 OpenClaw）因合规/安全原因被禁用，内部平台又过于技术化。
4. **跨工具协作 friction**：Excel 数据 -> Word 报告 -> PPT 演示 -> 飞书表格同步，链条长、手动操作多、容易出错。

解决思路：**用自然语言替代复杂菜单操作**，让全行员工（无论技术背景）都能通过说话/打字完成文档处理、数据分析和格式设置。

## Demand Evidence
- 领导指派调研开发任务，目标覆盖全行办公场景
- 内部类似 Dify 的 Agent 编排平台学习成本高，业务人员难以使用
- 外部 AI 桌面应用（如 OpenClaw）因合规/安全原因被禁用
- 业务人员目前处于"原始工作状态"：Excel 手动处理、Word 排版反复调格式、PPT 逐页制作、找 IT 写脚本
- 飞书多维表格推广受阻：很多员工反馈"功能太多学不会""不知道怎么设置自动化提醒""视图不会配"
- 日常办公中存在大量跨文档重复劳动：Excel 数据复制到 Word/PPT、手动调整格式、飞书表格和本地 Excel 来回同步

## Status Quo
业务人员目前通过以下方式完成办公文档和数据处理工作：
1. **Excel**：手动复制粘贴、写公式、做透视表、调格式，复杂操作（如条件格式、数据验证）需要搜索教程
2. **Word**：反复调整字体/段落/页眉页脚，制作目录需要手动设置大纲级别，排版耗时占文档工作 30% 以上
3. **PPT**：逐页设计版式、调配色、插入图表，从数据到演示文稿全靠手工
4. **飞书多维表格**：因功能入口深、字段类型多、视图配置复杂，多数员工仅当普通表格使用，自动化规则、分组视图等高级功能利用率低
5. **跨工具协作**：Excel 数据 -> 复制粘贴到 Word/PPT -> 手动调格式 -> 飞书同步，链条长、版本混乱
6. **提需求给 IT/开发同事**：排期等待，沟通成本高，小需求（如"帮我把这个表按部门汇总"）也占用开发资源
7. **行内简单 AI 问答或 AI 插件辅助**：只能对话，无法直接操作文件和格式
8. **内部 Dify-like 平台**：功能完整但工作流编排学习成本极高，无法自然语言处理所有事物

## Target User & Narrowest Wedge
目标用户为**全行员工**，覆盖所有办公场景下的非技术人员，无需编程或复杂配置能力即可上手。

第一批具体用户建议从"每月需要处理大量 Excel/CSV 报表的职能岗位"切入（如销售运营、财务分析、HR、运营支持）。

最小楔形（MVP）：
拖入一份 CSV/Excel/Word/PPT/飞书文档 -> 选择对应"专家模式"（数据分析/文档排版/演示文稿美化） -> AI 给出处理方案（Plan 模式）-> 用户点击同意 -> AI 调用工具并生成报告或修改后的文档 -> 右侧边栏实时出现产物并提供预览/下载。

## Constraints
- 内网环境，无法访问互联网 AI 服务，需对接内部大模型 API
- 安全要求高：需要本地工作区物理隔离、权限审批、操作留痕。审计日志写入本地 SQLite（`%APPDATA%/ONE/audit.db`），保留 90 天，记录每次工具调用的用户、时间、输入摘要、输出摘要，不包含原始文件内容。
- **数据合规与脱敏（底线要求）**：业务人员拖入的 Excel/CSV 文件中极大概率包含客户姓名、身份证号、手机号、银行卡号、员工薪资等敏感 PII。Electron 客户端不得将这些明文数据直接打包发给大模型——必须通过后端 BFF 网关的 Data Masking 组件进行正则脱敏（如手机号 → `138****8888`）后再转发。任何触发内网 DLP 审计的明文数据外泄风险都将导致项目被合规部门叫停。
- **SQLite 自动清理**：审计数据库启用 `PRAGMA auto_vacuum = INCREMENTAL`，让 SQLite 在日常运行中逐步回收碎片空间，避免整表重建。超过 90 天的旧记录清理时机安排在**应用退出时（`before-quit` 事件）或系统闲置时（通过 `powerMonitor` 监听 `idle` 状态）**，绝不在冷启动时执行，防止 `VACUUM` 导致的整表锁死和初始化卡顿。
- Windows 桌面应用（.exe 安装包）
- 业务人员非技术人员，不能暴露 Bash/文件路径等技术细节

## Resilience & Error Handling
- 内部 LLM API 临时不可用时：本地排队 + 指数退避重试（最多 3 次），UI 显示"模型服务繁忙，正在重试..."
- 完全离线场景：缓存最近的 system prompt 和工具定义，允许用户查看历史会话，但禁用新工具调用
  - **双轨离线检测策略**：
    - **后台心跳（30秒轮询）**：UtilityProcess 每 30 秒发送一次 LLM API health check（HEAD 请求），3 秒无响应即标记为离线，通过 `MessagePort` 通知渲染进程更新顶部横幅并禁用输入框。这是"预警"机制，让用户尽早知道网络异常。
    - **发送前检查**：用户每次点击发送前，UtilityProcess 再执行一次即时 health check。若此时发现离线，阻止本次请求并 toast 提示"网络不可用，请检查连接"。这是"拦截"机制，避免消息进入本地队列后长时间等待。
  - 离线 UI 状态：输入框禁用，顶部显示"网络不可用，仅支持查看历史会话"横幅，提供"重试"按钮
- 用户-facing 错误消息统一走中文语义化翻译层（位于渲染进程，UtilityProcess 传递原始 error code + args，React 组件负责映射到中文文案），不暴露原始 API 错误码

## Premises
1. 基于 free-code CLI 核心进行 Electron 包装和工具扩展，比从零开发更快，且能保留团队对技术栈的熟悉度。
1. **内部 LLM API 为 Anthropic 原生协议**：桌面端直接构造 Anthropic `/v1/messages` 格式请求体（原生 SSE，保留 `tool_use`/`tool_result` block 结构）。对 OpenAI/MiniMax 等模型的兼容交由后端 AI 网关（如 LiteLLM）处理，客户端不做协议翻译。
2. 交互复杂度的主要来源是"让业务人员理解 AI 在做什么"，因此 UI 层需要语义化翻译（把 Read/Write 工具调用翻译成"正在分析数据...""正在生成报告..."）。
3. 参考 claw-code 的架构设计（Plan 模式协议、McpToolRegistry、PermissionEnforcer）可以大幅降低 free-code 内核改造的风险。
4. Electron 的打包体积（>100MB）和内存占用在目标企业场景中可接受。
5. 领导指派的调研任务需要在 1-2 周内看到可演示的成果，才能争取后续资源。
6. **自然语言是最高效的 UI**：对于非技术用户，自然语言描述的交互效率远高于学习复杂菜单和多步向导，因此核心功能（特别是飞书多维表格配置、Word 排版、PPT 生成）应优先以对话式交互实现。
7. **全行用户的技能差异大**：从完全不懂 Excel 公式的行政人员到熟练使用透视表的财务分析师都是目标用户，AI 需要能自适应用户表达水平（用户说"帮我算一下总和"和"生成 SUMIF 公式"都应能处理）。

## Approaches Considered

### Approach A: free-code 源码改造 + Bun（构建工具）+ Electron（推荐）
- **直接 fork free-code 源码**，保留其 LLM API 客户端、对话循环、状态管理、命令/工具注册表等核心能力
- 剥离 CLI 入口（`src/entrypoints/cli.tsx`）和 Ink TUI 组件，替换为 Electron 主进程（窗口管理）+ UtilityProcess（AI 引擎）+ React 渲染进程
- 使用 **Bun 作为包管理器和构建工具**，Electron 主进程和渲染进程严格使用内置 Node.js 运行时
- 以 **claw-code 为架构参考**（Plan 模式、权限分级、MCP 设计），以 **cc-haha 为桌面交互参考**（Tauri 桌面实现、窗口管理、文件拖放）
- 工作量：5-7 天（Phase 0-1），代码基础扎实，后续扩展风险低

### Approach B: claw-code + Tauri（内核现成，技术栈切换）
- 利用 claw-code 已有的 Plan 模式、权限系统、任务管理，Tauri 与 Rust 原生集成
- 安装包 <5MB，但团队需具备 Rust 能力，与现有 TypeScript 生态割裂
- 项目无 LICENSE 文件，存在版权风险；且 claw-code 是 CLI 工具，改造为桌面应用的工作量被低估

### Approach C: cc-haha 二次开发（桌面现成，范围过大）
- cc-haha 已是一款完整桌面应用（Tauri + Ink TUI + Local Bun server + IM adapters），功能覆盖广
- 但体量过大（~643K LOC，112 命令，59 工具），大量功能（IM 适配、多运行时桥接）与当前需求无关
- 裁剪和理解的成本高于基于 free-code 增量开发

## Recommended Approach
**Approach A：fork free-code 源码，Bun 运行时，Electron 包装器。**

### 核心策略
| 维度 | 决策 | 理由 |
|---|---|---|
| 内核来源 | 直接修改 free-code 源码 | free-code 保留了最精简的 CLI AI 核心（API 客户端、对话循环、状态管理、工具注册表），与需求高度匹配 |
| 运行时 | Node.js（Electron 内置），Bun 作为包管理器和构建工具 | Electron 主进程深度绑定 Node.js/V8，无法替换为 Bun；Bun 用于依赖管理和构建加速 |
| 桌面壳 | Electron v28+ | 成熟、文档完善、Windows 企业场景验证充分 |
| 架构参考 | claw-code（Plan 模式、PermissionEnforcer、MCP 注册表） | 不导入代码，仅参考协议设计和分级策略 |
| 交互参考 | cc-haha（桌面窗口管理、文件拖放、侧边栏产物预览） | 已验证的桌面端交互模式，降低 UI 设计风险 |
| 协议风险 | 当前处于调研测试阶段，后续敏感时做修改 | 先验证可行性，再处理合规 |
| 后端策略 | **BFF AI 代理网关** | Electron 客户端不直接对接任何底层业务系统（飞书私有化 API、内部 OA 等），由后端团队提供统一的 BFF (Backend for Frontend) AI 代理网关。桌面端只负责构建 JSON 指令并发给网关，复杂的鉴权、路由转换、飞书 API 组装全部由后端微服务处理。这符合企业内网安全架构规范，也避免前端边界过度膨胀 |

### free-code 内核改造映射表
| free-code 原模块 | 改造后用途 | 迁移方式 |
|---|---|---|
| `src/services/api/client.ts` | LLM API 客户端（SSE 流式、多提供商适配） | **保留**，移至 UtilityProcess |
| `src/state/AppStateStore.ts` | 全局状态管理（对话历史、当前会话、配置） | **保留**，通过 `MessagePort` 暴露给渲染进程 |
| `src/commands.ts` | 命令注册表（`/` 斜杠命令） | **保留**，UI 层映射为下拉菜单/快捷键 |
| `src/tools/` | 工具执行引擎（文件操作、Bash、搜索等） | **保留**，UtilityProcess 执行，权限拦截器前置 |
| `src/services/permissions/` | 权限策略（如存在） | **保留**，扩展为三级分级（ReadOnly/WorkspaceWrite/DangerFullAccess） |
| `src/entrypoints/cli.tsx` | CLI 入口 | **删除**，替换为 Electron 主进程入口（窗口管理）与 UtilityProcess 入口（AI 引擎） |
| `src/components/`（Ink） | TUI 组件 | **删除**，替换为 React + Ant Design |
| `src/screens/` | TUI 屏幕 | **删除**，替换为 Electron 窗口 + React 路由 |
| `src/bridge/` | 远程桥接 | **暂不引入**，MVP 纯本地运行 |

### 风险与备用方案
- **若内部 LLM 不支持 tool calling**：降级为 prompt-based tool simulation（ReAct 风格），预计增加 2-3 天工作量
- **若 litellm 代理对图片/文档传输支持不佳**：绕过代理直接对接底层模型 API，或在源码层实现分块传输和 Base64 内联编码，预计增加 1-2 天工作量

### 核心模块职责

**Electron Main Process（主进程）**
- **职责**：纯窗口管理与系统服务。不做任何 LLM 请求、不做文档解析、不运行工具逻辑。
- **具体工作**：创建 BrowserWindow、管理应用生命周期、设置系统托盘/菜单、通过 `MessageChannelMain` 在 Renderer 和 UtilityProcess 之间转发端口。

**Electron UtilityProcess（AI Agent 引擎进程）**
- **职责**：承载整个 free-code 核心与 AI 逻辑。这是 Electron 为后台任务量身定制的独立 Node.js 进程（VS Code Extension Host 同款方案）。
- **关键优势**：① SSE 长连接或 500MB CSV 解析不会导致 UI 卡顿；② 引擎崩溃（OOM / 未捕获异常）只会杀死 UtilityProcess，主进程和渲染进程存活，可自动拉起新引擎。

1. **`api/client.ts`（保留改造）**
   - 原职责：与 Anthropic / Bedrock / Vertex 通信，处理 SSE 流式响应
   - 新职责：源码级支持 Anthropic 原生 `/v1/messages` 协议（保留 `tool_use`/`tool_result` block 结构）。Phase 0-1 直接对接内部 LLM API（通过后端 AI 网关如 LiteLLM 做协议转换）；后续可无缝切换到后端 BFF 网关的 LLM Proxy 接口，前端只需更换 `baseUrl`
   - **运行位置：UtilityProcess**（SSE 长连接和 JSON 解析不会阻塞主进程 UI）

2. **`AppStateStore.ts`（保留改造）**
   - 原职责：管理 CLI 会话状态（消息历史、当前会话 ID、配置）
   - 新职责：增加 `electron-store` 持久化层，启动时加载最近 20 个会话元数据
   - **同步方式**：状态变更后通过 `MessagePort` 推送至渲染进程（绕过主进程）

3. **`tools/`（保留扩展）**
   - 原职责：Bash、文件读写、搜索等工具执行
   - 新职责：增加文档处理工具（ReadExcel、FormatWord、GeneratePPT 等），Phase 2+ 实现
   - 权限控制：执行前经过 `PermissionService` 拦截器，写操作触发 Plan 模式审批
   - **运行位置：UtilityProcess**（工具执行不会阻塞 UI）

4. **`commands.ts`（保留映射）**
   - 原职责：`/` 斜杠命令注册与分发（如 `/compact`、`/cost`）
   - 新职责：命令触发方式从 CLI 输入映射为 UI 交互（下拉菜单、快捷键、右键菜单）
   - 新增命令：`/switch-model`（切换模型）、`/new-session`（新建会话）

5. **Backend BFF AI Gateway（后端，已确认）**
   - **LLM Proxy**：接收 Anthropic 原生 `/v1/messages` 格式请求，转发至内部 LLM（litellm/自研网关做协议转换），SSE 流式透传回桌面端。后期接管前端直接对接底层模型的职责
   - **Auth Service**：统一鉴权托管（JWT 签发与刷新、SSO 对接），桌面端无需保存飞书/OA 的原始 Token
   - **Feishu/OA Router**：接收桌面端发来的结构化 JSON 指令，翻译成飞书私有化 API 调用或内部 OA 接口调用，返回标准化响应。Phase 4 启用
   - **Data Masking / DLP 拦截**：在将 Prompt（含文件内容摘要）转发给底层大模型之前，必须经过脱敏组件。使用正则表达式匹配并替换敏感 PII：手机号 → `138****8888`、身份证号 → `110**********1234`、银行卡号 → `6222****1234`、薪资/金额等。这是金融企业强监管环境的底线要求，防止明文敏感信息流入大模型触发 DLP 审计
   - 位置：后端微服务，与 **UtilityProcess** 通过内网 HTTP 通信

### IPC 协议设计

采用 **双通道 IPC 架构**：Renderer 与 UtilityProcess 通过 `MessageChannelMain` 建立直连端口（高频率 SSE 流式通信不走主进程）；Renderer 与 Main Process 仅保留系统级操作（窗口控制、安全存储、文件读取）。

**通道一：Renderer ↔ UtilityProcess（MessagePort 直连）**

```typescript
// Renderer → UtilityProcess
interface EngineRequest {
  id: string;
  type: 'chat:send' | 'chat:cancel' | 'chat:retry'
       | 'plan:approve' | 'plan:reject'
       | 'workspace:create' | 'workspace:switch' | 'workspace:delete'
       | 'file:drop' | 'file:request' | 'config:update';
  payload: unknown;
}

// UtilityProcess → Renderer（流式 / 推送）
interface EngineEvent {
  id: string;
  type: 'chat:chunk' | 'chat:end' | 'chat:error'
       | 'tool:status' | 'plan:awaiting_approval' | 'plan:executing'
       | 'plan:completed' | 'plan:failed'
       | 'state:sync' | 'artifact:update' | 'offline:changed' | 'engine:crashed';
  payload: unknown;
}
```

**通道二：Renderer ↔ Main Process（contextBridge，仅限系统级操作）**

```typescript
interface MainAPI {
  // 文件系统：主进程读取文件，返回 buffer + 元数据（渲染进程不接触路径）
  readFile: (id: string) => Promise<{ name: string; size: number; buffer: ArrayBuffer }>;
  // 安全存储 API Key
  getEncryptedKey: (service: string) => Promise<string | null>;
  setEncryptedKey: (service: string, key: string) => Promise<void>;
  // 窗口控制
  minimize: () => void;
  maximize: () => void;
  close: () => void;
  // 系统目录
  getAppPath: (name: 'userData' | 'temp' | 'desktop') => string;
}
```

## Document Processing Capabilities（文档处理功能矩阵）

本应用定位为**全行办公人员的 AI 文档助手**，覆盖日常办公中高频使用的全部文档类型。核心设计原则是：**用户不需要学习复杂的产品功能，用自然语言描述需求，AI 自动完成格式设置和数据处理。**

### 支持格式与场景

| 文档类型 | 支持场景 | 自然语言交互示例 |
|---|---|---|
| **Excel / CSV** | 数据清洗、透视分析、公式生成、图表插入、格式美化 | "把 A 列的日期统一成 2024-01-01 格式" / "给销售额大于 10 万的行标红" / "生成按区域汇总的数据透视表" |
| **Word** | 排版美化、样式统一、目录生成、批量替换、格式调整 | "把标题改成黑体二号加粗，正文改成宋体小四" / "在第二页插入目录" / "所有一级标题前面加项目符号" |
| **PPT** | 模板选择、内容填充、数据图表插入、格式套用 | "根据这份 Excel 数据生成汇报 PPT，用标准汇报模板" / "把销售额趋势图插到第二页" / "给每页加上部门 logo 和页码" |
| **Markdown** | 转 Word/PPT/Excel、格式标准化、批量处理 | "把这个 Markdown 转成带样式的 Word 文档" / "把里面所有表格抽出来生成单独的 Excel" |
| **飞书多维表格** | 数据推送、字段映射、视图配置 | "把这个 Excel 的数据推送到飞书项目跟进表" / "给截止日期快到的任务标红" / "按负责人分组，看每个人的任务数量" |

### 核心卖点：零学习成本的智能文档操控

1. **飞书多维表格自然语言配置**
   - 用户无需学习飞书多维表格的字段类型（单选/多选/公式/关联）、视图切换、自动化规则等概念。
   - 直接描述业务需求，AI 自动翻译为底层字段配置和规则。例如：
     - 用户说："我想看这个表每个人手上有多少未完成的任务"
     - AI 自动：创建「负责人」分组视图 + 状态筛选（未完成）+ 计数统计
   - 支持一键回滚：AI 修改多维表格配置前自动创建字段级快照，用户可在侧边栏点击"恢复修改前状态"。

2. **Word 自然语言排版**
   - 用户无需知道"样式"、"母版"、"段落间距"等专业术语。
   - 用日常语言描述想要的视觉效果，AI 自动应用对应格式。例如：
     - 用户说："让这份报告看起来正式一点，像行里发的红头文件"
     - AI 自动：应用标准公文格式（标题居中、正文首行缩进、页眉加机构名称）
   - 支持"格式刷"语义化："把这一页的排版风格应用到整个文档"

3. **PPT 智能内容填充与模板套用**
   - 用户从预置模板列表中选择（如"标准汇报模板"、"年度总结模板"），AI 自动将生成的大纲和数据填入对应占位符。
   - 图表从原始数据自动生成（柱状图、折线图、饼图），插入到模板的图表页。
   - 例如："根据这份 Excel 数据生成汇报 PPT，用标准汇报模板" -> AI 自动：套用模板 -> 填入标题/正文 -> 生成销售额趋势图 -> 输出 .pptx
   - MVP 阶段图片不支持自动生成，留白或由用户后续手动插入。

4. **Excel 智能分析与报表**
   - 在现有 MVP 基础上扩展：不仅生成报告，还能直接修改原文件的格式和内容。
   - 例如："帮我把这个表的重复客户合并，保留最新联系时间" -> AI 自动去重、更新、高亮变更行。

### 飞书集成技术要点（BFF 网关架构）
- **部署形态**：内网采购的是私有化部署版本，非飞书官方 SaaS。API 规格需向 IT 部门确认，不能硬编码 `open.feishu.cn`。
- **核心原则**：Electron 客户端**绝不直接对接**飞书私有化 API。所有与飞书的通信必须通过后端团队提供的 **BFF AI 代理网关** 转发。桌面端只负责将用户意图转换为结构化 JSON 指令（如 `{ action: 'createView', tableId: 'xxx', config: {...} }`）发给网关，复杂的 OAuth 鉴权刷新、API 路由转换、错误码翻译、速率限制全部由后端网关处理。
- **架构策略**：
  - 桌面端预留 `IExternalTableProvider` 接口，但底层实现不是直接 HTTP 请求，而是向 BFF 网关发起 IPC → UtilityProcess HTTP 调用。
  - MVP 实现 `MockTableProvider` 用于演示（本地模拟网关响应）。真实网关对接在 Phase 4 完成，通过配置切换 `baseUrl` 从 mock 地址到真实 BFF 网关地址。
- **MVP Demo 方案**：AI 处理本地 Excel/CSV → 生成结构化 JSON → 调用 `MockTableProvider`（模拟 BFF 网关响应）→ UI 显示"已推送至内部表格系统"。这能证明"具备打通能力"，不阻塞 MVP。
- **鉴权（后续）**：飞书私有化 API 的鉴权 Token（OAuth2 / API Key / SSO Token）**由后端 BFF 网关统一托管和刷新**，不落地 Electron 客户端。桌面端仅持有向 BFF 网关鉴权的短期 Token（如 JWT），由后端统一签发和轮换。
- **安全合规**：外部系统 token 不保存在桌面端；审计日志仅记录"向 BFF 网关发起推送指令"的行为摘要，不包含飞书原始 Token。

### PPT 模板策略
- **模板来源**：安装包内置 3-5 个行内标准模板（.pptx 文件），用户从下拉菜单选择。模板每年更替时通过应用更新分发。
- **AI 职责**：不是"设计师"，而是"填表员"——解析模板结构（标题页、正文页、图表页），将生成内容填入占位符。
- **图表生成**：用 `pptxgenjs` 从数据自动生成柱状图/折线图/饼图，插入模板图表页。
- **图片处理**：MVP 阶段不支持 AI 生成图片，留白或由用户手动插入。后续可接入内部图库或 AI 生图服务。
- **扩展路径**：Phase 4 后支持用户上传自定义模板，AI 自动识别占位符（如 `{{title}}`）。

   - **Plan 拒绝后的恢复路径**：用户点击"拒绝"后，PlanCard 不消失，而是在卡片下方原地展开一个微调输入框，预填用户原始需求，允许用户修改后点击"重新生成计划"。降低非技术用户的心智负担，避免重新复制粘贴长串需求。
   - Plan 执行前快照（基于时间戳的版本快照）：在当前 Workspace 沙盒目录中建立隐藏文件夹 `.snapshots/`。每次 Agent 在 Plan 模式下获得授权准备修改文件前，UtilityProcess 生成一个带版本游标的物理拷贝，例如 `v0_1000_data.csv`（初始）、`v1_1005_data.csv`（第一次修改前）、`v2_1010_data.csv`（第二次修改前）。回滚时定位到对应版本复制并强制覆盖工作区文件。
   - **Windows 文件锁（File Lock）处理**：快照覆盖前需显式捕获 `EBUSY` 错误。若文件正被 Excel/WPS 占用，通过语义化 UI 提示用户："请先在外部关闭该文件后再试"，而不是暴露原始错误码。
   - MVP 初版可简化为单步备份（保留 `v0` 原始版本 + `v_latest` 最新版本），接口预留版本号字典，后续可无缝升级到多版本历史时间线。
   - 防膨胀机制（GC）：UtilityProcess 封装垃圾回收逻辑，仅保留最近 5 次修改记录，超出的物理文件自动 `fs.unlink` 删除；超过 50MB 的文件不纳入快照，仅记录路径引用。
   - 参考 claw-code 的 `EnterPlanMode/ExitPlanMode` 和 `PermissionEnforcer` 设计。
4. **Phase 3（Week 3）—— 业务工具（五大格式）**：硬编码实现 ReadExcelTool、GenerateReportTool、ReadWordTool、FormatWordTool、ReadPPTTool、GeneratePPTTool、FeishuBitableTool。接入产物管理侧边栏（基于 `chokidar` 实时同步）。专家 Prompt 路由扩展为"数据分析师"、"文档排版师"、"演示文稿设计师"、"飞书表格管理员"。MCP 标准化接口以 stub 形式预留（定义 `ITool` 接口但不实现动态加载），注册表延后到第五个工具需求出现时再实现。
5. **Phase 4（Week 4）—— 端到端贯通**：核心链路测试扩展为"五大格式主链路测试"（Excel 分析 -> Word 排版 -> PPT 生成 -> Markdown 转换 -> 飞书表格配置）。SSO 登录作为可选加分项，如内部 SSO 对接复杂，先用本地 API key 或简单账号stub 完成试点。

### IPC & State Synchronization
- **Renderer ↔ UtilityProcess 直连**：主进程在启动时创建 `MessageChannelMain`，将其中一个端口通过 `contextBridge` 交给渲染进程，另一个端口随 `utilityProcess.fork()` 传给引擎进程。此后高频率通信（SSE 流式 token、状态同步、工具状态）直接通过 `MessagePort` 传输，**不经过主进程**，消除主进程转发瓶颈。
- **Renderer ↔ Main Process（系统级）**：仅用于窗口控制、文件系统读取（返回 `ArrayBuffer` + 元数据，不暴露路径）、`safeStorage` 加密存储。通过 `contextBridge.exposeInMainWorld` 暴露，配合 `.d.ts` 类型声明实现编译时安全。
- **状态同步策略**：UtilityProcess 维护唯一的 free-code 核心实例（`AppStateStore` + `ConversationLoop`），渲染进程只持有 UI 状态的只读副本；所有状态变更由 UtilityProcess 驱动，通过 `MessagePort` 推送至渲染进程。
- **产物监听**：UtilityProcess 对当前 Workspace 目录挂载 `chokidar`，文件变化后通过 `MessagePort` 推送至 React 侧边栏。

### 文件拖放交互规范
- 拖放区：React 主面板支持 `onDrop`，拖拽进入时高亮边框；同时支持粘贴飞书文档链接（自动识别 `https://*.feishu.cn/...` 格式）
- 文件验证：接受 `.csv`、`.xlsx`、`.xls`、`.docx`、`.pptx`、`.md`，单文件大小上限 50MB，超出时 toast 提示"文件过大，请拆分后上传"
- 飞书链接处理：UtilityProcess 通过 BFF 网关获取表格/文档元数据（字段结构、记录摘要），将结构信息注入 LLM context，不在 UI 暴露原始 token
- **路径处理**：主进程读取文件后通过 IPC 将内容传给 UtilityProcess，由 UtilityProcess 提取摘要（前 100 行或前 10 页幻灯片大纲）并注入 LLM context，全程不在 UI 暴露原始绝对路径
- **数据脱敏声明**：首次启动时，聊天界面顶部显示一次性横幅提示："请勿上传包含客户敏感信息的明文文件，系统将自动进行数据脱敏。" 用户勾选"我已了解"后方可继续使用。每次拖入文件时，底部再次出现微提示："文件内容将脱敏后发送至 AI 模型处理。"
- **脱敏链路说明**：文件内容摘要（前 100 行）由 UtilityProcess 提取后，通过 HTTP 发送至后端 BFF 网关。BFF 网关在转发给底层大模型之前，必须经过 Data Masking 组件对敏感 PII（手机号、身份证号、银行卡号、薪资等）进行正则替换脱敏。这是金融企业强监管环境的底线要求。

### Build & Development Workflow
- **构建工具链**：使用 `electron-vite` 作为构建工具（Vite 负责 React 渲染进程 HMR，Electron 主进程通过 `vite-plugin-electron` 热重载）。Bun 作为包管理器（`bun install` 替代 `npm install`），利用其 lockfile 和安装速度优势。
- **开发阶段热重载**：
  - 渲染进程：`vite` 原生 HMR，React 组件修改秒级刷新。
  - 主进程：`vite-plugin-electron` 监听主进程文件变更后自动重启 Electron。
- **打包与分发**：
  - 使用 `electron-builder` 配置 Windows NSIS 安装包和 Portable 便携版。
  - 代码签名：内网企业环境需准备 Windows 代码签名证书（或 IT 提供内部信任链签名），否则杀毒软件可能误报。
- **内网自动更新**：Electron 默认 `autoUpdater` 依赖 GitHub Releases 或自定义更新服务器。内网无法访问公网时，需搭建内网更新服务器（如基于 Express 的静态文件服务 + 版本 JSON 清单），主进程启动时拉取 `latest.yml` 或自定义版本文件比对，下载差量包后调用 `autoUpdater` 本地安装。

## Appendix: MVP 策略讨论记录（2026-05-11）

本次讨论聚焦于 Phase 3 "业务工具扩展" 中 PPT 模板和飞书集成的最小可行性策略。

### 已确认的前提

| 问题 | 结论 | 影响 |
|------|------|------|
| PPT 模板来源 | 有推荐模板，范围有限（3-5 个），不会自由发挥；但模板可能每年更替，部门可能有定制化 | MVP 内置标准模板 + 支持选择；后续支持用户上传 |
| PPT 图片处理 | 模板内图片靠自己发挥 | MVP 阶段不支持 AI 生成图片，留白或占位；图表用数据可视化库自动生成 |
| 飞书部署形态 | 私有化部署，非飞书官方 SaaS，名字/图标均不同 | 不能硬编码 `open.feishu.cn`，必须预留抽象接口 |
| 飞书 MVP 目标 | 预留 API 接入口子，Demo 展示"具备打通能力" | 用 MockProvider 演示，真实 API 延后 |

### 选定方案

**PPT: Approach A — 预置模板 + AI 填表**
- 安装包内置 3-5 个标准模板，用户从下拉菜单选择
- AI 解析模板结构，填入生成内容
- 图表用 `pptxgenjs` 从数据自动生成
- 图片暂不支持（MVP 留白）
- **工作量**: 1-2 天

**飞书: Approach A — 抽象接口 + Mock 演示**
- 预留 `IExternalTableProvider` 接口
- MVP 实现 `MockTableProvider`，模拟 API 响应
- Demo 展示：AI 处理 Excel → 生成 JSON → Mock 推送 → UI 显示"已推送"
- 真实 API 对接延后到 Phase 4
- **工作量**: 1-2 天

### 关键决策

1. **PPT 不做"AI 设计师"，做"AI 填表员"** — 模板固定，AI 只负责内容填充和格式套用，大幅降低实现复杂度。
2. **飞书不做"全自动双向同步"，做"能力证明"** — 通过抽象接口 + Mock 演示证明"我们能打通"，不阻塞 MVP 交付。
3. **图片/图表分离处理** — 图表由数据可视化库生成，图片由用户手动插入，MVP 阶段不碰 AI 生图。

- [ ] 向 IT 部门确认私有化部署的 API 文档、网关地址和鉴权方式（Phase 4 阻塞项）
