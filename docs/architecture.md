## 1. Architecture Overview

### 1.2 Why UtilityProcess

| Concern | Main-Process-Only | Worker Threads | Hidden Renderer | **UtilityProcess** |
|---|---|---|---|---|
| UI blocking risk | High | Medium (GC still in main) | Medium | **Eliminated** |
| Crash isolation | None | Partial | Good | **Full (VS Code mode)** |
| Node.js API access | Full | Full | Limited | **Full** |
| Startup overhead | None | Low | Medium | **Low** |
| IPC complexity | Minimal | Medium | Medium | **Medium (structured clone)** |
| Dev/debug experience | Simple | Medium | Hard | **Simple (Node.js process)** |

**Key advantage:** If the AI engine OOMs on a 500MB malformed CSV, only the UtilityProcess dies. The main process catches `utilityProcess.on('exit')`, shows a toast "AI engine crashed, restarting...", and respawns. The React UI never drops a frame.

## 2. Module Responsibilities

### 2.1 Main Process (`src/main/`)

| Module | Responsibility |
|---|---|
| `main.ts` | App lifecycle, single-instance lock, window creation, menu/tray |
| `window-manager.ts` | BrowserWindow factory, responsive breakpoint enforcement, theme sync |
| `ipc-router.ts` | Forwards `MessagePort` between Renderer and UtilityProcess; handles system-level IPC (file-open, deep-link) |
| `protocol-handlers.ts` | Register `one://` protocol for internal resources |

**Critical rule:** Main process never touches LLM APIs, never parses documents, never runs tool logic. It is a pure router and system coordinator.

## 3. IPC Protocol

### 3.1 Renderer ↔ UtilityProcess (via MessageChannelMain)

The main process creates a `MessageChannelMain`, passes one port to the renderer via `contextBridge`, and the other port to the UtilityProcess on spawn. After setup, messages flow directly without touching the main process.

```typescript
// src/shared/ipc-types.ts

// Renderer → Engine
interface EngineRequest {
  id: string;
  type:
    | 'chat:send'           // Send user message
    | 'chat:cancel'         // Abort current generation
    | 'chat:retry'          // Retry last failed message
    | 'plan:approve'        // Approve pending plan
    | 'plan:reject'         // Reject pending plan + provide feedback
    | 'workspace:create'    // Create new workspace
    | 'workspace:switch'    // Switch active workspace
    | 'workspace:delete'    // Delete workspace
    | 'file:drop'           // User dropped file(s)
    | 'file:request'        // Request file preview
    | 'config:update';      // Update settings (baseUrl, model, theme)
  payload: unknown;
}

// Engine → Renderer (streaming / push)
interface EngineEvent {
  id: string;
  type:
    | 'chat:chunk'          // SSE token chunk
    | 'chat:end'            // Stream complete
    | 'chat:error'          // LLM error with structured code
    | 'tool:status'         // Tool execution step update
    | 'plan:awaiting_approval'  // Plan card should show
    | 'plan:executing'      // Plan execution started
    | 'plan:completed'      // Plan done
    | 'plan:failed'         // Plan execution failed
    | 'state:sync'          // Full state snapshot (workspace list, messages)
    | 'artifact:update'     // New/modified file in workspace
    | 'offline:changed'     // Online/offline status changed
    | 'engine:crashed';     // UtilityProcess died, will respawn
  payload: unknown;
}
```

### 3.2 Renderer ↔ Main (contextBridge)

Used only for system-level operations that require main process privileges.

```typescript
// Preload exposes:
interface MainAPI {
  // File system: main process reads file, returns buffer + metadata
  // (Renderer NEVER gets file paths, only content + abstract IDs)
  readFile: (id: string) => Promise<{ name: string; size: number; buffer: ArrayBuffer }>;

  // Safe storage for API keys
  getEncryptedKey: (service: string) => Promise<string | null>;
  setEncryptedKey: (service: string, key: string) => Promise<void>;

  // Window controls
  minimize: () => void;
  maximize: () => void;
  close: () => void;

  // System info
  getAppPath: (name: 'userData' | 'temp' | 'desktop') => string;
}
```

### 3.3 Data Flow: Send Message

```
User types + Enter
    │
    ▼
Renderer: UnifiedConsole.onSubmit()
    │
    ▼
window.engineAPI.postMessage({ type: 'chat:send', payload: { content, files } })
    │
    ▼ (MessagePort — structured clone)
UtilityProcess: ConversationLoop.handleUserMessage()
    │
    ├── Build context (system prompt + history + file summaries)
    ├── Apply client-side PII masking regex
    ├── POST /v1/chat/completions (SSE stream via axios)
    │
    ▼
LLM API → SSE chunks
    │
    ▼
UtilityProcess: parse SSE → emit 'chat:chunk' events
    │
    ▼ (MessagePort)
Renderer: MessageList receives chunk → append to state → React re-render
    │
    ▼
If tool call detected:
    UtilityProcess: PermissionService.check(tool)
        ├── ReadOnly → execute immediately, emit 'tool:status'
        ├── WorkspaceWrite → PlanStateMachine.transition(AWAITING_APPROVAL)
        │                    emit 'plan:awaiting_approval'
        └── DangerFullAccess → same as WorkspaceWrite (MVP)
```

## 4. State Machines

### 4.1 Plan Mode State Machine

```
                    ┌─────────────┐
         ┌─────────►│    IDLE     │◄────────────────────────────┐
         │          └──────┬──────┘                             │
         │                 │ user sends message                 │
         │                 ▼                                    │
         │          ┌─────────────┐      timeout/abort         │
         │    ┌────►│  THINKING   │────────────────────────────┘
         │    │     └──────┬──────┘
         │    │            │ LLM decides tool needed
         │    │            ▼
         │    │     ┌─────────────┐
         │    └─────┤  PLANNING   │◄───────────────────────────┐
         │          └──────┬──────┘                            │
         │                 │ plan JSON ready                   │
         │                 ▼                                   │
         │          ┌─────────────┐     user clicks reject     │
         │          │ AWAITING    │────────────────────────────┘
         │          │ APPROVAL    │     (with retry text)
         │          └──────┬──────┘
         │                 │ user clicks approve
         │                 ▼
         │          ┌─────────────┐     error during exec      │
         │          │  EXECUTING  │────────────────────────────┘
         │          └──────┬──────┘
         │                 │ all tools done
         │                 ▼
         │          ┌─────────────┐
         └──────────┤  COMPLETED  │
                    └─────────────┘
```

**Single-thread rule:** When in `AWAITING_APPROVAL`, the input box is disabled. No new messages can be sent until the user approves or rejects.

### 4.2 Offline Detection State Machine

```
         ┌──────────┐
    ┌────►│  ONLINE  │◄────────────────────────────┐
    │     └────┬─────┘                             │
    │          │ 30s heartbeat fails OR            │
    │          │ pre-send health check fails       │
    │          ▼                                   │
    │     ┌──────────┐      user clicks retry      │
    └─────┤ OFFLINE  │─────────────────────────────┘
          └──────────┘
```

**Dual-track strategy:**
- **Background heartbeat:** Every 30s, HEAD request to `${baseUrl}/health`. 3s timeout → mark offline.
- **Pre-send check:** Before every `chat:send`, instantaneous health check. If offline → block request, toast "网络不可用".

## 10. Performance Considerations

### 10.1 Memory

| Concern | Mitigation |
|---|---|
| Large file upload (50MB) | Stream file content, never load entire buffer into renderer |
| SSE accumulation | `eventsource-parser` yields chunks incrementally; renderer appends to React state |
| Long conversation history | Load metadata only; full messages on demand; 20-turn auto-compaction |
| Snapshot GC | Keep max 5 versions; skip files >50MB |
| SQLite audit | `auto_vacuum = INCREMENTAL`; cleanup on exit, not startup |

### 10.2 CPU

| Concern | Mitigation |
|---|---|
| Document parsing (Excel/Word) | Runs in UtilityProcess, never blocks UI |
| JSON deserialization | Large LLM responses chunked; parse incrementally |
| Markdown rendering | `react-markdown` with `max-h-[60vh]`; virtualize if >100 messages |

### 10.3 Network

| Concern | Mitigation |
|---|---|
| SSE long connection | Silent reconnect on abrupt termination, preserve context |
| API timeout | Configurable `timeoutMs`, default 30s |
| Retry storm | Exponential backoff (1s→2s→4s), max 3 retries |
| Offline detection | Dual-track: 30s heartbeat + pre-send check |

## 11. Failure Modes & Mitigations

| Scenario | Detection | Handling | User Impact |
|---|---|---|---|
| LLM API 5xx/timeout | HTTP status / axios timeout | Retry 3× with backoff → error message in chat | "遇到了一点小问题，正在重试..." + 重试按钮 |
| SSE stream abort | `eventsource-parser` EOF unexpectedly | Silent reconnect, preserve tokens | 无感知 |
| UtilityProcess crash | `utilityProcess.on('exit')` | Respawn engine, sync state, toast "AI引擎重启中" | 短暂提示后恢复 |
| 500MB malformed CSV | Tool executor catch | OOM kills UtilityProcess only, main/renderer survive | "文件解析失败，请检查格式" |
| File locked by Excel | `fs.copyFile` throws `EBUSY` | Catch, semantic toast | "请在外部关闭该文件后重试" |
| Path traversal attack | `workspace/validator.ts` regex | Reject with audit log | Silent prevention (security) |
| SQLite DB corruption | `better-sqlite3` open fail | Recreate DB, log event | 审计记录重置，应用继续运行 |
| Network offline | Heartbeat / pre-send fail | Disable input, show banner | 顶部红色横幅 + 禁用发送 |
| Config parse error | `electron-store` JSON parse fail | Fallback to defaults, log warning | 设置重置为默认 |
| PII regex miss | DLP audit on BFF gateway | Gateway catches residual PII | 合规拦截（最后防线） |

## 12. Security Architecture

### 12.1 Data Flow Security

```
User drops Excel file
    │
    ▼
Main Process reads file → passes ArrayBuffer to Renderer (no path exposed)
    │
    ▼
Renderer sends file content to UtilityProcess via MessagePort
    │
    ▼
UtilityProcess extracts summary (first 100 rows)
    ├── Client-side PII regex masking (first defense)
    ├── HTTP POST to BFF Gateway
    │
    ▼
BFF Gateway Data Masking component (second defense)
    │
    ▼
Internal LLM API
```

### 12.2 Key Security Measures

1. **No raw file paths in renderer.** Main process abstracts files as `{ id, name, size, buffer }`.
2. **API keys in safeStorage.** Never written to plaintext config.
3. **Workspace sandbox.** All file operations confined to `%APPDATA%/ONE/workspaces/<uuid>`.
4. **Path traversal prevention.** `workspace/validator.ts` rejects `../` and absolute paths.
5. **Double-layer PII masking.** Client-side regex + BFF gateway regex before LLM.
6. **Audit everything.** SQLite logs every tool call with user, time, input/output summaries (no raw file content).
7. **No direct Feishu API calls.** All external system access routed through BFF gateway.
8. **contextIsolation: true, nodeIntegration: false.** Renderer cannot access Node.js APIs.

## 16. Architecture Decisions

| Decision | Context | Rationale |
|---|---|---|
| UtilityProcess over main-process-only | User requirement: eliminate UI jank, achieve VS Code-level crash isolation | Full Node.js API access, true process isolation, modern Electron best practice |
| MessageChannelMain over `ipcMain`/`ipcRenderer` | Need direct Renderer ↔ Engine communication without main-process bottleneck | Zero-main-process overhead for high-frequency SSE streaming |
| `better-sqlite3` over `sqlite3` (async) | Audit logging is write-heavy and synchronous-by-nature | Simpler code, no callback hell, acceptable for local DB |
| Tailwind v3 over v4 | Team familiarity, stable plugin ecosystem (`@tailwindcss/typography`) | v4 is new; v3 has proven ecosystem for our needs |
| shadcn/ui minimal usage | Most components are custom chat UI, not standard forms | Only Settings Dialog uses shadcn; everything else is custom Tailwind |
| Client-side PII masking + BFF gateway masking | Defense in depth for financial compliance | First line catches obvious patterns; second line (gateway) is authoritative |
| Single-thread Plan (AWAITING_APPROVAL blocks input) | Prevent non-technical users from getting confused by multiple pending plans | Simpler mental model; revisit in V1.2 if power users request multi-plan |
| `electron-store` over custom JSON files | Encrypted-at-rest, schema migration support, atomic writes | Better than hand-rolled file I/O for config/session metadata |
| NSIS + Portable over MSI | Faster install, no Windows Installer service dependency, easier CI | Bank IT can push portable version for quick pilot |

## 17. Open Engineering Questions

1. **UtilityProcess debugging:** VS Code debugger configuration for attaching to UtilityProcess (`runtimeExecutable` + `address: "localhost:9223"`). Need to verify this works with `electron-vite` HMR.
2. **MessageChannel structured clone limits:** File content (ArrayBuffer) passing through MessagePort — verify 50MB buffer transfer performance vs shared memory.
3. **`better-sqlite3` native module rebuild:** This package has native bindings. Need to ensure `electron-builder` rebuilds it correctly for Windows target on macOS dev machine (or use GitHub Actions Windows runner).
4. **PPT template parsing:** `pptxgenjs` can generate but not parse existing templates. We may need `pptx-parser` or manual XML traversal to identify placeholders. This needs a 1-day spike before Phase 3.
5. **Windows code signing certificate:** IT department timeline for providing certificate — gates NSIS installer distribution (portable can be unsigned for pilot).
6. **UtilityProcess respawn state recovery:** If engine crashes mid-conversation, how much state is lost? `AppStateStore` persists to disk every N seconds — need to define checkpoint frequency.
